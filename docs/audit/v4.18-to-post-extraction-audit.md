# Post-Extraction Audit: v4.18 → `439579a`

**Audience:** a fresh Claude instance or human reviewer, booted with no prior context about the extraction work.

**Your role:** the bulldog. Your job is to find bugs, question design choices, evaluate correctness, **and honestly measure whether this investment paid off** — not to ship more code, not to propose more refactors. Every finding you report should be **actionable** (fix it, file an issue, or justify why the current code is right). Every "win" you claim should be backed by specific evidence.

**Do not trust** any prior claim that "this was reviewed" or "CI is green." CI green means *it compiles and the tests pass*; this audit is about whether the tests themselves are right, whether subtle behaviors were preserved, whether the design holds up to scrutiny, and whether the codebase is genuinely more defensible now than at v4.18.

---

## Context in 200 words

Between `v4.18` (tag) and `439579a` (master HEAD as of 2026-04-23), the OnSpeed Gen3 firmware underwent a major refactor: most platform-independent logic was extracted from the sketch into `software/Libraries/onspeed_core/`. Driving goal was to enable a second hardware target (Gen2v4) to be added as a new `HardwareMap.h` + `.ino` + symlink, with zero duplication of flight-critical logic.

**19 squash-merged PRs, 219 files changed, +21,365 / −4,394 lines.** The list of PRs is in the "What changed" section below; each is a self-contained commit on master.

**No firmware from this stack has flown yet.** The plan was always to bundle Phase 3 + 4 into one flight test after a bench-test shakedown. Your audit informs that bench-test prep.

**The stakes.** This is an experimental general-aviation AOA indicator. The audio tones the pilot hears directly drive stall-avoidance and approach-speed cues. A subtle numeric regression in AHRS or audio could move the stall warning to the wrong side of reality. Unit tests did not exist for most of the extracted code before this work started; we wrote ~500 new tests as we went. That is both the big win and the big risk: many tests pin the implementation we wrote, not the implementation master had before — if we ported a bug exactly, the tests now prevent it from ever being noticed.

## Your mission, concretely

**Four** evidence-gathering passes. The first measures the ROI of the refactor so Sam can present a credible case to other developers. The next three are critique. Budget roughly equal effort across all four — Pass 0 is not a throwaway intro.

### Pass 0: Investment ROI — was it worth it?

The refactor was 19 PRs over ~2 weeks, +21,365/-4,394 lines across 219 files. Sam's contention is that this makes the codebase **easier to find bugs in and more resilient** than v4.18. Your job is to test that contention with evidence, not take his word.

Concrete questions to answer with specific numbers and examples:

1. **How much of the +21k is tests vs production code?** Break it down by directory:
   - `test/**` — new test files. How many tests? How many lines?
   - `software/Libraries/onspeed_core/src/**` — new core code. How much is extraction (moved from sketch) vs genuinely new (schema, tripwires, etc.)?
   - `software/sketch_common/src/**` — sketch-side. How many lines moved from the sketch root? How many net-new?
   - `docs/**` — documentation additions.
   - `scripts/**`, `.github/workflows/**` — CI infrastructure.

   Don't just count — compare line-for-line. `git diff --stat v4.18..origin/master -- path/` gives you the stat; follow up with a sample read of 3-5 files to confirm the nature of the change (genuine new logic vs mechanical move).

2. **Test coverage: before vs after.** v4.18 had (approximately) 136 tests per the v4.17 release notes. HEAD has 565. What does this actually buy:
   - Which core modules gained test coverage they didn't have before? (EKF6, Madgwick, Kalman, EMA filters, SavGol derivative, tone calculation, sensor conversions, EFIS parsers, log CSV, config XML round-trip, etc.)
   - For each, estimate the **bug class** the tests guard against. A test that pins `(parent, tag)` pairs for WebSchema guards against silent XML-tag rename. A test that round-trips `n720ak_2_11_26.cfg` guards against silent config-format drift. Concrete, not generic.
   - Are any tests **tautological** (compute expected value the same way the function does)? Those don't catch bugs. Flag them.
   - What did CI **not** have before that it has now? `check_core_purity.sh`, `check_board_flags.sh`, snapshot regression, docs completeness. For each, what class of bug would it have caught if it had existed during a specific v4.17/v4.18 bug report? (Sam has git history of bugs; `git log --oneline --grep="fix" v4.17..v4.18` is a starting point.)

3. **The "easier to find bugs" claim — strongest evidence available.** Sam's contention is that the new structure makes bug-finding easier. **You have direct evidence for this: `docs/audit/README.md` and its 53 numbered findings from a comprehensive pre-extraction audit on 2026-04-08.** Walk through the severity-ranked list (Critical, High, Medium, Low sections). For each of the **sketch-side 20 findings** (001-020 — the Gen3 firmware ones, not the M5-prefixed ones), answer:

   - Where did the bug live on master (as of audit-date `b45909e`)?
   - Where does the equivalent code live now (post-extraction)?
   - Would the bug **still exist** in some form (not yet fixed), or is it impossible-by-construction in the new structure?
   - Crucially: **could a bulldog run over the new code find this bug faster than they could on v4.18**? A unit test that reproduces the bug in 1 minute vs hours of instrumenting firmware is a huge signal.
   - For bugs whose fix requires firmware flash + bench test on master, does the new structure allow a host-native reproduction?

   This is the single strongest data point for Sam's argument. Don't hand-wave — fill out a table with 20 rows. If even half of those 20 are easier to find/fix now, that's a concrete win. If most aren't, the argument has to be narrower.

   Additionally skim the 33 M5-prefixed findings (021-053) — they're out of the extraction's main scope but they tell you whether the structure at least *doesn't make them harder*.

   Also cross-reference the GitHub issues tracker (`#82, #93, #102, #116, #128, #131, #146, #149, #150, #164`) for bugs found after v4.17 — same analysis applies.

4. **The "more resilient" claim.** Resilience = the code keeps working when surroundings change. Evidence:
   - PR #211 (the M5GFX `fonts::` fix) — would v4.18 have caught this breakage? Under what CI coverage?
   - The core-purity CI gate — was there ever a pre-extraction situation where platform code leaked into otherwise-portable logic? (Hard to evidence backwards; look at the file move diffs for hints — cases where a `#include <Arduino.h>` was removed from an extracted module.)
   - The snapshot regression harness — even with its known limitations (issue #204), it provides drift detection that didn't exist before. Quantify: if you intentionally change one line in `Ahrs::Step` (e.g., alter a filter constant by 1%), does the harness catch it? Do the unit tests catch it?

5. **Honest losses / costs.** What did we **give up** or complicate? Be specific:
   - Build complexity: `sketch_common/` symlink adds a Windows gotcha. PR 4.1's WebSchema is mostly decorative today (the renderer isn't routed). The codecov thresholds and cppcheck recursion were tweaked and re-tweaked.
   - Reviewer load: every PR was a line-by-line review; subtle bugs were caught and fixed (PStaticBias sign, GLimit `>=` regression, AHRS `Init()` TAS reset, WebSchema xmlTag mismatches, docs-completeness leaf collision). Those were close calls. What does that say about the risk of extraction in general?
   - Test maintenance: 565 tests will need to keep passing as new code lands. Some test fixtures (the N720AK config, sample log row) need to stay in sync with the struct. Is that realistic?

6. **Size up the case for other developers.** Given all of the above, produce a **one-page "executive summary" table** with columns: *Claimed win*, *Evidence*, *Caveat*, *Rating* (Strong / Moderate / Weak / Not established). Sam will use this exact table when presenting the work. Do not pad it with generic claims — every row should stand up to cross-examination.

If Pass 0 concludes that some of the claimed wins aren't well-supported by evidence, say so in plain language. Sam would rather present an honest "modest win" than an unsupported "huge win."

### Pass 1: Behavioral-equivalence audit (the most important)

For **every flight-critical module** extracted, diff it against `v4.18` and verify byte-for-byte behavioral equivalence — or find a specific numerical divergence.

Flight-critical modules (in decreasing order of risk):

1. **AHRS pipeline** (`software/Libraries/onspeed_core/src/ahrs/Ahrs.cpp` + `sketch_common/src/tasks/AHRS.cpp`) vs v4.18's `software/OnSpeed-Gen3-ESP32/AHRS.cpp`. This drives every audio tone, every liveview field, every log column. The most dangerous regression. A line-by-line walk is the right level of care.

   - Compare constants, branch order, sign conventions, isnan/isinf guards, operation grouping (FP is non-commutative), EMA seeding, gyro-bias initialization, Kalman altitude/VSI flow, TAS derivative, FlightPath, DerivedAOA (`pitch - flightpath` Madgwick-only), EarthVerticalG, accel correction (Fwd/Lat/Vert).
   - The snapshot regression harness (`tools/regression/run_snapshot.py`) has a known limitation: it's a bit-stability gate, not a cross-version gate. The golden was regenerated when AHRS moved to core because the previous harness (PR 0.3) didn't exercise AHRS. Issue #204 tracks making the harness physically meaningful. **Do not trust the snapshot as proof of equivalence.**
   - Reviewer during extraction already flagged one real divergence (`Ahrs::Init()` was resetting TAS state that legacy didn't) — see commit history on master if you want the pattern. Look for similar.

2. **Audio synthesis** (`onspeed_core/src/audio/ToneSynth.cpp`, `WavDecode.cpp`, `AudioMixer.cpp`) vs v4.18's `Audio.cpp`. What the pilot hears.

   - Critical to verify: legacy `uint16_t(25000.0f * cosf(...))` cast produces bit-identical output to the new `SynthesizeLegacyCosine` function. Verify at IEEE-754 level for the actual firmware amplitudes (25000) and frequencies (400, 1600 Hz). Reviewer claimed this was verified; do your own check.
   - Verify voice-gain cap-at-1.0 ordering preserved (audit #019 — legacy has a cap that must not shift order).
   - Verify pulse-counter state preservation across call boundaries.

3. **EFIS parsers** (`onspeed_core/src/efis/*.cpp`) vs v4.18's `EfisSerial.cpp`. Six vendor protocols.

   - The extraction introduced a NaN sentinel design (`EfisFrame` default = NaN) to fix a silent "hold last value" regression in the first implementation. Reviewer walked the math; do your own on at least one parser (Dynon SkyView recommended since Sam flies VN-300 and Dynon).
   - D10 Palt/VSI alternating-frame bug — verify the fix is really fixing it and not re-introducing the zeroing behavior.

4. **Config** (`onspeed_core/src/config/*.cpp`) vs v4.18's `Config.cpp`. XML parse/emit + V1 CSV legacy + defaults.

   - Reviewer found and fixed one Critical: `GetString` was clearing field to `""` on empty XML tag; legacy left it untouched (letting defaults survive). Verify the fix is still in place and comprehensive.
   - Audit #009 (PStaticBias V1↔V2 sign convention) is pinned by `test_v1_v2_pstatic_normalize_to_same_memory`. Read the test and confirm the pin is correct.
   - Audit #013 (unbounded flap loop): new parser truncates rather than rejects. Behavior is arguably correct but different from master — audit this.
   - Look for any intentional-but-undocumented behavior change.

5. **Housekeeping decisions** (`onspeed_core/src/audio/GLimitDecision.cpp`, `VnoChimeDecision.cpp`, `VolumeCurve.cpp`) vs v4.18's `Housekeeping.cpp`.

   - Reviewer caught `>` vs `>=` boundary regression in G-limit (now fixed); verify.
   - Cooldown semantics: master used integer tick counter (100ms tick), new code uses `tickMs` ms-diff. Verify the math produces the same wall-clock behavior.

6. **Log CSV** (`onspeed_core/src/proto/LogCsv.cpp`) vs v4.18's `LogSensor.cpp` / `LogReplay.cpp`.

   - The log format is load-bearing for downstream tools. Verify byte-identity via the test fixtures.
   - Look for Issue #193 (empty-field-as-zero) and Issue #194 (vnTimeUtc comma) risks; these were acknowledged not-fixed.

### Pass 2: Design review

Independent of bugs, examine the design choices for each layer. Look for:

- **Over-abstraction.** Is there a schema / layer / interface that doesn't pay for itself with a real consumer? The `WebSchema` specifically — does it earn its keep given it only has two real consumers (a tripwire test and an un-routed renderer)? (See below for pre-existing concern.)
- **Under-specification.** Are there POD types (`EfisFrame`, `AhrsOutputs`, `LogRow`) where a field's units, range, or semantics aren't clear from the declaration? A reader of the core types alone should be able to use them correctly.
- **Leaky boundaries.** Does anything in `onspeed_core/src/` reach into globals, Arduino types, or FreeRTOS? The `check_core_purity.sh` script catches header includes but not *indirect* leaks (e.g., a function that assumes a specific call frequency without saying so).
- **Duplicated helpers.** `SafeScaledInt`/`ClampInt` exist in both `DisplaySerial.cpp` (sketch) and `proto/DisplaySerial.cpp` (core) — is that the only duplication, or are there more? (Issue #192 tracks the DisplaySerial one.)
- **sketch_common/ symlink risk.** A Windows contributor without `core.symlinks=true` would see a 20-byte text stub and the build would fail loudly. Is the documentation in `docs/site/docs/software/building.md` enough? Is there any protection against a symlink loop or inadvertent `sketch_common` edit that sidesteps the shared review pattern?
- **Test coverage shape.** Is there a module in core with < 70% line coverage where a real bug could hide? Run coverage with `pio test -e native-coverage` + lcov.

## Ground rules for the bulldog

1. **Do not write code in this pass.** File issues, don't land PRs. The purpose is analysis, not rework.
2. **Cite file:line for every finding.** Claims like "AHRS has a bug in the TAS block" are useless without "line 187 uses `.prev` before the first-frame seed."
3. **Use subagents for parallel depth**, but verify their claims before including them in the report. Prior reviews have been wrong in ways that mattered (see commit message on PR 3.2's review-fix commit — a reviewer claimed `fAlpha0Floor` was already in master when they hadn't checked; the actual bug was different).
4. **Flight test is the load-bearing gate, not you.** Your job is to make that flight test as informed as possible, not to replace it. Tell Sam what to watch for in flight and on the bench.
5. **Do your own line-by-line on AHRS specifically.** Everything else can be delegated to subagents. AHRS is too flight-critical to trust a single reviewer on.

6. **On Pass 0 — be honest, not a cheerleader.** Sam wants a credible case to present to other developers, not a marketing doc. If a claimed win turns out to be thin, write that. The value of the report to Sam is inversely proportional to how much of it is generic "improved maintainability" hand-waving.
