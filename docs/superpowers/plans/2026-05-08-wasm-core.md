## The thesis

> Compile `software/Libraries/onspeed_core/` to WebAssembly. The
> Replay tool's JS layer loads the WASM module and calls into it
> for every algorithm. **Zero hand-ports.** No drift between
> firmware C++ and Replay JS — by construction, they ARE the same
> code.

The architecture collapses:

```
                          ┌──────────────────────────────┐
                          │   software/Libraries/        │
                          │   onspeed_core/              │  ← THE SPEC
                          │   (platform-free C++)        │
                          └──────────────┬───────────────┘
                                         │
              ┌────────────┬─────────────┼────────────┬───────────────┐
              │            │             │            │               │
              ▼            ▼             ▼            ▼               ▼
        Firmware       X-Plane      M5 simulator   host_main    onspeed_core
        (C++ on        plugin       (full-featured CLI binary    .wasm
         ESP32)        (C++ on      WASM, browser) (subprocess  (algorithm-only
                       Mac/Win/                     wrappers     WASM, browser)
                       Linux)                       use this)         │
                                                                       │
                                                                       ▼
                                                            ┌──────────────────┐
                                                            │  Replay tool     │
                                                            │  (browser)       │
                                                            │                  │
                                                            │  JS UI calls     │
                                                            │  into WASM for   │
                                                            │  every algorithm │
                                                            └──────────────────┘
```

**Three of the four browser/desktop targets become WASM/native
compiled C++.** Python wraps host_main. Only the M5 hardware variants
(Basic, Core2, huVVer-AVI) are runtime-distinct, and they only render
the firmware's wire output — no algorithm code on those boxes.

**No hand-ports anywhere.** The architecture is: write the algorithm
in C++ once, ship it to every target.

## Why this is the right move

1. **Drift is impossible by construction.** No streaming-goldens CI
   gate, no JS-vs-C++ mismatch concerns, no "did I forget to update
   the JS port?" tax on every C++ change.
2. **Future algorithm work is free in JS.** Sam adds spin-recovery
   cue logic in C++ → next replay deploy has it. No port effort.
3. **Performance is fine.** Algorithms are arithmetic-only (no
   allocations in hot path, fixed-size structs). FFI overhead per
   call is ~50-100 ns when warm; per-frame load at 50 Hz is
   nanoseconds. Bundle size: ~200-400 KB WASM, fine for
   `dev.flyonspeed.org/replay`.
4. **The build infrastructure already exists.** The M5 WASM
   simulator (`software/OnSpeed-M5-Display/sim/`) compiles M5GFX +
   `onspeed_core` to WASM with Emscripten. We're doing a SUBSET of
   that work — algorithm-only, no rendering. Strictly easier.

## What this does NOT do

- **Does NOT replace the M5 simulator.** That's the full M5
  firmware in WASM (with M5GFX rendering). It stays as the docs-site
  preview. This new WASM build is algorithm-only — no renderer.
- **Does NOT replace the firmware.** Firmware is still
  ESP32-native C++ via PlatformIO.
- **Does NOT eliminate `host_main`.** host_main remains for Python
  subprocess wrappers + CI tests. It's the same C++; just compiled
  for native-CLI use instead of WASM browser-use.
- **Does NOT eliminate JS for the Replay tool.** The Preact UI
  components stay JS. WASM is for the **algorithm calls only**:
  percent-lift, anchors, EMA, lever sweep, config parse, etc.

## Why this beats the streaming-goldens CI gate

Earlier plan: "JS hand-port + CI gate against host_main output."

This plan: "JS calls into WASM-compiled C++; no port to gate."

The CI gate was protection against drift. With WASM, **there's no
drift to protect against** — JS literally invokes the same compiled
algorithm the firmware runs. We get the same correctness guarantee
with less infrastructure.

## What we're giving up

Three real costs:

1. **Build complexity**: emcc in CI, WASM module artifacts in
   deploy, slightly more involved local-dev setup. ~3-5 days of
   one-time work.
2. **Bundle size**: +200-400 KB WASM module on the Replay page
   load. Vac downloads it once and the browser caches it. For a
   bookmarked desktop tool this is fine.
3. **Debug experience**: WASM bugs show up as "RuntimeError:
   memory access out of bounds" with poor source mapping. Mitigated
   by debugging at the C++ side via host_main subprocess (Python
   wrappers exercise the same code).

Sam's prior M5-sim experience flagged memory issues. **Important
distinction**: those were almost certainly the M5GFX renderer (M5
allocates framebuffers, font caches, sprite caches). The
algorithm-only WASM build is much smaller — no rendering, just
arithmetic. Empirically: bundle ~200-400 KB, runtime heap ~few MB.

## Sequenced rollout

Each step independently shippable.

### Step 0 — Build infrastructure (~1-2 days)

What to ship:

1. **`software/Libraries/onspeed_core/wasm/build_wasm.sh`** — emcc
   build script that:
   - Compiles every `.cpp` under `onspeed_core/src/`
   - Excludes anything that depends on Arduino / FreeRTOS (the
     `check_core_purity.sh` script already enforces this — leverage)
   - Outputs `onspeed_core.js` + `onspeed_core.wasm` (~200-400 KB
     each)
   - Uses `-O3 -s MODULARIZE=1 -s EXPORT_ES6=1
     -s ALLOW_MEMORY_GROWTH=1`
   - Optional: `-s SINGLE_FILE=1` to inline WASM into the JS for
     simpler deploys (one fewer artifact to host)

2. **A wrapper module** at
   `software/Libraries/onspeed_core/wasm/bindings.cpp` — exposes
   the public C++ API to JS via `EMSCRIPTEN_KEEPALIVE` or
   `embind`. Choose embind for cleaner typing.

3. **CI step** in `.github/workflows/docs.yml` (since the WASM
   ships in the dev.flyonspeed.org deploy): install emcc, run the
   build, copy artifacts into the static deploy.

4. **Dev-server integration**: dev-server serves the WASM artifact
   from the build output. Local rebuild: `bash build_wasm.sh`.

**Test gate**: a tiny JS file that loads the module and calls one
function (e.g. `computePercentLift(3.24, ...)` returns ~49.6).
Land in `tools/web/test/wasm-smoke.mjs`, plumbed into `npm test`.

### Step 1 — Bind percent-lift + config + anchors (~1 day)

What to ship:

1. **`bindings.cpp` exposes**:
   - `compute_percent_lift(aoa_deg, alpha_0, alpha_stall, stallwarn_aoa) -> float`
   - `compute_anchors(active_flap_setpoints_struct, all_flaps_array, raw_adc) -> AnchorsStruct`
   - `parse_config(xml_text) -> ConfigStruct`

2. **JS wrappers** at `tools/web/lib/replay/wasm_core.js`:
   ```js
   import OnSpeedCoreModule from '/static/onspeed_core.js';
   let Core = null;
   export async function initCore() {
     if (Core) return Core;
     Core = await OnSpeedCoreModule();
     return Core;
   }
   export function computePercentLift(aoaDeg, flapCfg) {
     return Core._computePercentLift(...);
   }
   ```

3. **Replace the JS ports** at:
   - `lib/replay/percentLift.js` → 20-line wrapper around WASM
   - `lib/replay/config.js` → 30-line wrapper around WASM

4. **Run the existing render-smoke + replay tests**. Should pass
   without changes (same outputs, different implementation).

### Step 2 — Bind log-replay pipeline (~1-2 days)

**Depends on `PLAN_FIRMWARE_LOG_REPLAY_PARITY.md` having landed.**
That plan fixes two firmware-side gaps the JS Replay tool currently
patches over: (a) IMU EMA rate-coupling when the log is at 50 Hz
instead of the firmware's 208 Hz, and (b) synthesizing flap-pot ADC
sweep when the log lacks the `flapsRawADC` column. The WASM
LogReplay binding is meant to be the **same code path** the
firmware's LogReplay mode runs — if those gaps are still in JS,
this step would either move the JS hack into a WASM-bound
replay-only wrapper (which diverges from firmware) or skip the
behavior entirely (broken replay). Land the firmware fix first.

The big one. What to ship:

1. **`bindings.cpp` exposes the streaming pipeline**:
   - `replay_init(config_struct) -> ReplayHandle`
   - `replay_step(handle, raw_row_struct) -> RecordStruct`
   - `replay_destroy(handle)`
   - The handle holds EMA state, lever-sweep state, etc. across
     calls — same lifetime semantics as the firmware's
     long-running task loop.
   - Internally, this binds the same `LogReplay` C++ that the
     firmware's LogReplay mode runs — including the rate-correct
     ingest and synth-ADC fixes from PLAN_FIRMWARE_LOG_REPLAY_PARITY.

2. **JS wrapper** at `lib/replay/wasm_replay.js`. Allocates a
   handle on first use; calls `replay_step` per CSV row; frees the
   handle on log change.

3. **Replace** `lib/replay/logReplay.js` — becomes a wrapper that
   calls into the WASM streaming pipeline. The current
   `KACC_TAU_S = 0.50` variable-dt EMA loop and `synthLeverSweep`
   two-pass smoothstep code in JS are **deleted** (~80 LOC). The
   C++ in firmware LogReplay does both, correctly, and the WASM
   build inherits it.

4. **Pre-pass over the entire log**: today the JS replay
   pre-computes per-row smoothed accels + per-row synth ADC up
   front. Match this in WASM by walking the log once at load time,
   storing the results in a Float32Array on the JS side. Lookup
   stays an O(1) array index per video frame.

### Step 3 — Bind data-mark detection + sync detection (~half-day)

Smaller helpers but worth porting so JS doesn't reimplement:

1. **`bindings.cpp` exposes**:

   - `detect_first_crosswind(log_struct) -> int`
   - `find_data_marks(log_struct) -> array of MarkStruct`

2. **JS wrappers** replace `lib/replay/syncDetect.js` and

The streaming-goldens CI gate from `PLAN_DRIFT_PREVENTION.md`
exists to catch JS-vs-C++ drift. With WASM, **there is no drift to
catch**. Remove the JS regression CI job. Keep the C++
host_main snapshot regression — it still catches behavior changes
that need careful review.

`PLAN_DRIFT_PREVENTION.md` collapses to a one-paragraph note about
this transition.

## Bundle-size reality check

What's in the WASM module:
- `onspeed_core/aoa/` — percent-lift, anchors. Tiny.
- `onspeed_core/audio/` — ToneCalc + envelope. Larger but still small.
- `onspeed_core/ahrs/` — Madgwick fusion + EKF. Biggest single
  module; ~30 KB compiled.
- `onspeed_core/efis/` — VN-300, Dynon, etc. parsers. Medium.
- `onspeed_core/filters/` — EMA, Kalman, Savitzky-Golay. Tiny.
- `onspeed_core/proto/` — wire format builders. Small.
- `onspeed_core/sensors/` — Honeywell pressure conversions. Tiny.

Total compiled with `-O3` minus standard library noise: estimate
~300 KB WASM + ~150 KB JS shim. Within the order-of-magnitude
comfort zone for a desktop tool.

If we want to ship even smaller, we can build an algorithm-only
subset (just `aoa/` + `filters/` + the bits the Replay tool actually
calls). But starting with the full module is simpler and lets future
Replay features tap any onspeed_core capability without a
rebuild-and-redeploy.

## Memory: addressing Sam's concern

Sam mentioned the M5 simulator hit memory issues at realtime. The
algorithm-only WASM avoids the things that drove that:

- **No M5GFX framebuffer**: M5 sim allocates 320×240×4 = 307 KB per
  frame, plus sprite caches, plus font glyph caches. We don't.
- **No SDL2**: M5 sim uses SDL2 for canvas backing; cumulative
  memory cost. We don't render — we just compute.
- **No texture upload**: M5 sim copies the framebuffer to a WebGL
  texture every frame. We don't.

Algorithm-only WASM heap: ~few MB at most. The Replay tool's
biggest memory consumer remains the 17 GB video blob URL (handled
by the browser's media stack, not our code).

## Coordination with Python consolidation

Both plans require `host_main` to be a thing. The consolidation
plan's Step 0 ("multi-subcommand CLI") covers the host_main side;
the WASM plan's Step 0 covers the WASM build. They share the
binding work — `bindings.cpp` for WASM and `host_main` argparse
shells should expose the same C++ API surface, just with different
front-ends.

Reasonable sequence:
1. Python consolidation Step 0 (host_main multi-subcommand) — gets
   us argparse-style invocation of the C++ API.
2. WASM Step 0 (build infrastructure) — gets us the same C++ API
   accessible from JS.
3. Python consolidation Steps 1-7 in parallel with WASM Steps 1-5.
   Different teams or different sessions can work on these
   independently.

Both consolidate onto the same C++ API surface. After both land,
the only hand-port left in the system is **none**.
